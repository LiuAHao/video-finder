## 1. 优化目标

本轮优化的目标不是新增功能，而是把当前“网页视频发现与下载”主链路做稳，重点解决以下问题：

1. 候选资源中混入 HLS `.ts` 分片
2. “集数相关性”规则误伤真实视频资源
3. 下载输出路径缺少边界约束
4. 协议相对 URL（`//cdn...`）漏抓

优化原则：

- 优先修正行为错误，再调优排序体验
- 先收敛误判，再补测试
- 所有启发式规则都尽量“降级为惩罚”，不要轻易“直接过滤”

## 2. 执行顺序

建议分 4 个阶段执行：

### Phase 1：修复候选污染问题

- 目标：彻底阻止 `.ts` 分片进入候选列表
- 优先级：P1

### Phase 2：重构集数相关性逻辑

- 目标：减少误杀真实视频资源
- 优先级：P1

### Phase 3：修复下载路径边界问题

- 目标：保证下载文件落在预期目录内
- 优先级：P2

### Phase 4：补齐 URL 兼容性与回归测试

- 目标：减少漏抓并固化修复结果
- 优先级：P2

## 3. 详细优化步骤

## Phase 1：修复 HLS `.ts` 分片重新混入候选

### 问题描述

当前 `app/services/sniffer.py` 中：

- `_on_request()` 会过滤 `.ts` HLS 分片
- 但 `_on_response()` 只看 `content-type`
- 常见 `.ts` 分片响应类型是 `video/MP2T`
- 导致被请求层过滤掉的分片，在响应层又被重新加入 `_network_resources`

### 优化目标

- `.ts` 分片不能出现在最终候选列表中
- 请求层和响应层过滤规则保持一致

### 具体步骤

1. 修改 `app/services/sniffer.py`
   - 在 `_on_response()` 开头增加：
     - `if self._is_hls_segment(url): return`
   - 保证响应层与请求层一致过滤 HLS 分片

2. 增加第二层保护
   - 在 `_combine_candidates()` 中，网络资源转候选前再判断一次：
     - 若是 `.ts` 分片则跳过
   - 这样即使未来某处监听逻辑再变化，也不容易重新漏进来

3. 检查 content-type 策略
   - 当前 `VIDEO_CONTENT_TYPES` 包含 `video/MP2T`
   - 可以保留，用于识别流式资源
   - 但候选阶段必须结合 URL 规则，不可单靠 `content-type`

### 测试补充

新增/调整测试：

- 请求层过滤 `.ts` 分片
- 响应层遇到 `video/MP2T` 的 `.ts` URL 不应加入候选
- `m3u8` manifest 仍然正常保留

### 验收标准

- 候选列表不再出现 `.ts` 分片 URL
- HLS 页面仍可正常抓到 `.m3u8`

## Phase 2：重构“集数相关性”逻辑

### 问题描述

当前 `app/services/extractor.py` 的 `_extract_candidate_episode_hint()` 过于激进：

- 文件名尾部数字可能被直接当成“集数”
- `720p`、`1080p`、`001` 这类资源编号/清晰度会被误判
- 当前逻辑部分场景直接过滤候选，风险较高

### 优化目标

- 只在“强语义”模式下识别集数
- 不再因为普通数字误删真实候选
- 相关性判断主要用于排序，不轻易用于硬过滤

### 具体步骤

1. 收紧“页面集数”提取规则
   - 保留这些强模式：
     - `第01集`
     - `ep01`
     - `episode-01`
   - 对 `/1.html` 这种页面尾号模式保留支持
   - 但避免把更大范围的裸数字路径当成集数语义

2. 收紧“候选集数”提取规则
   - 删除或禁用这些弱模式：
     - 结尾任意数字
     - 任意下划线后数字
   - 只保留强语义模式：
     - `第01集`
     - `ep01`
     - `episode_01`

3. 调整 `_is_relevant_to_page()` 的使用方式
   - 对 HTML 静态提取：
     - 不再把“不相关”直接过滤掉
     - 改为允许进入候选，但降低评分
   - 例外：
     - 如果某类来源特别容易产生多集污染，可保留更强限制，但要加测试证明

4. 调整 `calculate_score()`
   - 将“疑似非当前集”只作为降分项
   - 降分幅度建议控制在 `10-20`
   - 不要直接掉到 0，避免真实资源完全消失

5. 增加更稳的相关性辅助信号
   - 如有标题时，可比对：
     - 页面标题
     - 资源 URL 中是否包含当前标题关键词
   - 这部分先作为可选增强，避免本轮改动过大

### 测试补充

新增测试用例：

- `episode_01.mp4` 在第 1 集页面保留
- `episode_02.mp4` 在第 1 集页面降分或靠后
- `video_720.mp4` 不应被当成“第 720 集”
- `source_1080.m3u8` 不应被误判为其他集
- `cdn_001.mp4` 不应因为弱尾号模式被硬过滤

### 验收标准

- 候选不再因为 `720/1080/001` 被误删
- 同时“明显是下一集/其他集”的资源不会排到推荐位

## Phase 3：修复下载输出路径边界问题

### 问题描述

当前 `app/web/routes.py` 中，`output_name` 直接拼接到 `download_dir`：

```python
Path(download_dir) / request.output_name
```

这会带来：

- 绝对路径绕过下载目录
- `../` 路径逃逸
- 文件落到不可预期位置

### 优化目标

- 所有下载输出默认受 `download_dir` 约束
- 自定义文件名只能影响“文件名”，不能控制目录层级

### 具体步骤

1. 提取路径安全辅助函数
   - 建议新增到：
     - `app/services/path_utils.py`
     - 或 `app/services/safety.py`
   - 函数示例：
     - `sanitize_output_name(name: str) -> str`
     - `build_safe_output_path(download_dir: str, output_name: str) -> Path`

2. 处理规则
   - 去掉目录成分，只保留 basename
   - 去除控制字符和非法文件名字符
   - 如果结果为空，回退默认命名
   - `resolve()` 后检查最终路径是否仍位于 `download_dir.resolve()` 之下

3. 在 Web 和 CLI 入口统一使用
   - `app/web/routes.py`
   - `app/cli.py` 中如果也支持自定义输出名，也要共用同一逻辑

### 测试补充

新增测试：

- 输入 `../../out.mp4` 最终仍落在下载目录
- 输入 `/tmp/out.mp4` 不应写到 `/tmp`
- 输入 `my-video.mp4` 正常生成
- 输入非法字符名称时会被清洗

### 验收标准

- 所有输出路径都位于配置下载目录内
- 自定义文件名仍可正常使用

## Phase 4：补齐协议相对 URL 支持与回归测试

### 问题描述

当前 `app/services/extractor.py` 中：

- `_extract_from_json_config()` 先判断 `_is_url(value)`
- `_is_url()` 仅支持 `http://` / `https://`
- 导致 `//cdn.example.com/video.m3u8` 这种协议相对 URL 被漏掉

### 优化目标

- 协议相对 URL 可以正常进入候选
- URL 判定逻辑统一交给 `_normalize_url()`

### 具体步骤

1. 调整 `_extract_from_json_config()`
   - 去掉 `_is_url(value)` 的前置限制
   - 改成：
     - 先 `url = self._normalize_url(value, frame_url)`
     - 再判断 `url` 是否存在、是否像媒体资源

2. 复核其他入口
   - `_extract_from_scripts()`
   - `_extract_from_data_attributes()`
   - `_extract_from_video_tags()`
   - 确保这些地方都先 normalize，再做媒体判断

3. 如果 `_is_url()` 只剩很少用途
   - 可考虑删除
   - 或改成支持 `//`

### 测试补充

新增测试：

- JSON config 中 `//cdn.example.com/video.m3u8` 能抓到
- `//cdn.example.com/video.mp4` 能抓到
- `blob:` / `data:` / `javascript:` 仍然被排除

### 验收标准

- 协议相对媒体 URL 被正常识别
- 非 HTTP 伪协议仍不会混入候选

## 4. 建议的实施顺序

为了减少一次性改动过大，建议按下面顺序执行：

1. `sniffer.py` 修 `.ts` 分片回流
2. `extractor.py` 调整集数识别规则
3. `routes.py` / 公共工具层修输出路径安全
4. `extractor.py` 补协议相对 URL 支持
5. 补单元测试与回归测试
6. 用 3-5 个真实页面做手动回归

---

## 5. 回归测试清单

每轮改动后建议至少执行：

### 自动化

- `venv/bin/python -m compileall -q app`
- `venv/bin/python -m pytest -q`

### 手动场景

1. 普通直链页面
   - 应抓到单个 mp4

2. HLS 页面
   - 应抓到 `.m3u8`
   - 不应出现 `.ts` 分片

3. 多集详情页
   - 当前集资源应排在前面
   - 其他集资源不应被推荐

4. 包含包装播放器地址的页面
   - `player.php?url=...` 不应排到前面

5. JSON 配置里使用 `//cdn...`
   - 应正常抓到资源

6. 自定义输出文件名下载
   - 文件仍落在下载目录

## 6. 风险与注意事项

### 风险 1：过滤规则过强

如果“集数相关性”仍然过于激进，可能继续误杀真实资源。

应对方式：

- 先改为“降分”而非“硬过滤”
- 先扩大保留，再通过排序改善体验

### 风险 2：过滤规则过弱

如果完全不做相关性控制，多集详情页仍然会很脏。

应对方式：

- 保留强语义识别
- 对不确定项仅轻度降分

### 风险 3：路径安全修复影响现有使用习惯

少数用户可能希望把文件直接下到任意目录。

应对方式：

- 默认强约束在下载目录
- 若后续确有需要，再单独设计“允许自定义完整输出路径”的高级选项

---

## 7. 最终交付标准

本轮优化完成后，应满足：

1. 候选列表基本不再出现 `.ts` 分片
2. 候选排序优先展示真实视频资源
3. 不再因为 `720p/1080p/001` 之类数字误删真实资源
4. 协议相对 URL 可正常识别
5. 自定义输出名不能逃逸下载目录
6. 自动化测试覆盖上述关键行为
